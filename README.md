# Gridfinity Bin Batcher

Current version: `v0.1.0`

Batch-generate Gridfinity Extended bins from CSV, wrap them as Bambu Studio
project files, and slice/package printer-specific Printago-ready
`.gcode.3mf` files.

This repo should stay focused on Gridfinity bins. The broader idea, turning
arbitrary STL/3MF files into Printago-ready plate-sliced packages, is captured
separately in `FUTURE_PRINTAGO_AUTOMATION.md`.

## Attribution

This project builds on
[ostat/gridfinity_extended_openscad](https://github.com/ostat/gridfinity_extended_openscad)
for the underlying Gridfinity Extended OpenSCAD models.

Gridfinity itself was created by Zack Freedman / Voidstar Lab. See
[zackfreedman.com](https://zackfreedman.com/) for the original Gridfinity work
and related projects.

## Current Status

The standard library has been generated and validated locally:

- `45` STL files
- `45` plain source 3MF files
- `45` Bambu project 3MF files
- `45` X1C `.gcode.3mf` packages
- `45` P2S `.gcode.3mf` packages

The Bambu project files use a 90 degree Z rotation and a 5 mm front purge-line
clearance. The final `.gcode.3mf` packages were audited so each packaged plate
metadata bbox starts at `Y ~= 5`.

## Output Layout

Generated files are organized under `exports/`:

```text
exports/
  stl/                 # OpenSCAD STL exports
  3mf/
    source/            # Plain OpenSCAD 3MF exports
    bambu/             # Bambu Studio project 3MF files
  gcode3mf/
    x1c/               # Printago-ready X1C .gcode.3mf packages
    p2s/               # Printago-ready P2S .gcode.3mf packages
```

Local working files live under `work/` when possible. Older scratch folders
such as `source-output/`, `bambu-output/`, `sliced-output/`, and `output/` are
not part of the final workflow.

## Printer Presets

This repo does not include printer preset JSON files. Each user must export or
copy their own Bambu Studio printer presets into:

```text
profiles/printer-presets/
```

Current expected preset files:

```text
profiles/printer-presets/Bambu Lab X1C 0.4 Large Part Cool 30º.json
profiles/printer-presets/Bambu Lab P2S 30º Large Part Eject Code.json
```

Bambu Studio CLI needs a machine-typed copy of each preset. Store those in:

```text
profiles/cli-printer-presets/
```

Current expected CLI preset files:

```text
profiles/cli-printer-presets/x1c_Bambu Lab X1C 0.4 Large Part Cool 30º.json
profiles/cli-printer-presets/p2s_Bambu Lab P2S 30º Large Part Eject Code.json
```

Each CLI preset JSON must include:

```json
{
  "type": "machine"
}
```

The original exported preset is still used when stamping printer metadata into
the final `.gcode.3mf` package.

These JSON files are intentionally ignored by Git because they are local Bambu
Studio exports. Keep the `.gitkeep` files so another user gets the expected
folder structure after cloning.

### Part-Ejection G-Code

Automated repeat printing requires a printer preset that already contains the
desired part-ejection G-code. This project does not generate, validate, or
install part-ejection machine code by itself.

For part-ejection setup, see the companion repo:
[baschwar/3D-Printed-Part-Ejection-GCode](https://github.com/baschwar/3D-Printed-Part-Ejection-GCode).

If you want Printago to run bins as unattended repeat jobs:

- Create or copy a Bambu Studio printer preset that includes your tested
  part-ejection start/end G-code.
- Export that preset JSON into `profiles/printer-presets/`.
- Create the matching CLI copy in `profiles/cli-printer-presets/`.
- Slice with that profile so the ejection behavior is embedded in the generated
  `Metadata/plate_1.gcode` inside each `.gcode.3mf`.

Part-ejection G-code is printer-, build-plate-, filament-, and part-specific.
Test it carefully before using it for unattended production.

## Bin Defaults

Only these values vary per CSV row:

- `grid_columns`
- `grid_rows`
- `height`

Project defaults live in `defaults.json`:

- Gridfinity base pitch: 42 mm
- Stacking lip style: minimum
- Wall thickness: 1.00 mm
- Magnets: disabled
- Screws: disabled
- Label style: disabled

All other settings come from upstream
`gridfinity_extended_openscad/gridfinity_basic_cup.scad`.

## Step-By-Step Batch Run

Run commands from the repository root:

```text
/path/to/Gridfinity-Bin-Batcher
```

### 1. Create The Bin CSV

For a custom run, edit `bins.csv`:

```csv
name,grid_columns,grid_rows,height
bin_1x3x6,1,3,6
```

For the standard 45-bin library:

```bash
python3 generate_permutation_csv.py
```

This writes:

```text
bins-unique-1x1-through-5x5.csv
```

The standard library includes unique `1x1` through `5x5` sizes at heights
`6`, `12`, and `18`, omitting rotational duplicates.

### 2. Generate STL Files

```bash
python3 generate_bins.py \
  --csv bins-unique-1x1-through-5x5.csv \
  --output-dir exports/stl \
  --formats stl \
  --openscad-prefix "arch -x86_64" \
  --openscad /Applications/OpenSCAD.app/Contents/MacOS/OpenSCAD
```

### 3. Generate Plain Source 3MF Files

```bash
python3 generate_bins.py \
  --csv bins-unique-1x1-through-5x5.csv \
  --output-dir exports/3mf/source \
  --formats 3mf \
  --openscad-prefix "arch -x86_64" \
  --openscad /Applications/OpenSCAD.app/Contents/MacOS/OpenSCAD
```

### 4. Create Bambu Project 3MF Files

The Bambu project template is:

```text
templates/bambu/gridfinity_bin_template.3mf
```

It is a manually saved Bambu Studio project with the desired project structure.
Do not overwrite it casually.

Wrap all source 3MF files into Bambu project 3MF files:

```bash
python3 batch_make_bambu_projects.py --force
```

Default output:

```text
exports/3mf/bambu/
```

Placement behavior:

- 256 mm x 256 mm plate assumption
- X centered
- front edge at Y = 5 mm
- Z rotation = 90 degrees by default
- placement math uses the rotated footprint, so the front edge stays at Y = 5 mm

### 5. Slice And Package For Printago

Slice all Bambu project 3MF files and package Printago-ready `.gcode.3mf`
files:

```bash
python3 batch_slice_gcode_3mf.py \
  --profile x1c \
  --profile p2s \
  --force \
  --continue-on-error
```

Defaults:

```text
input projects: exports/3mf/bambu/
output files:   exports/gcode3mf/
work files:     work/slicer/
```

The script is resumable. If `--force` is omitted, existing final packages are
skipped.

The generated `.gcode.3mf` files inherit machine behavior from the selected
Bambu printer preset. If the preset includes part-ejection G-code, the packaged
G-code will include it. If the preset does not include part-ejection G-code, the
packages will be normal sliced print files.

If Bambu Studio takes longer on a file, increase the timeout:

```bash
python3 batch_slice_gcode_3mf.py \
  --profile x1c \
  --profile p2s \
  --force \
  --continue-on-error \
  --slice-timeout 420
```

To repackage from existing sliced G-code without rerunning Bambu Studio:

```bash
python3 batch_slice_gcode_3mf.py \
  --profile x1c \
  --profile p2s \
  --package-only \
  --force \
  --continue-on-error
```

## Validation

Check expected counts:

```bash
find exports/stl -name '*.stl' | wc -l
find exports/3mf/source -name '*.3mf' | wc -l
find exports/3mf/bambu -name '*.3mf' | wc -l
find exports/gcode3mf/x1c -name '*.gcode.3mf' | wc -l
find exports/gcode3mf/p2s -name '*.gcode.3mf' | wc -l
```

For the standard batch, each count should be `45`.

Check every final package for the 5 mm front placement:

```bash
python3 -c "import json, zipfile; from pathlib import Path; bad=[]; checked=0
for p in sorted(Path('exports/gcode3mf').glob('*/*.gcode.3mf')):
    with zipfile.ZipFile(p) as z:
        d=json.loads(z.read('Metadata/plate_1.json'))
    y=d['bbox_all'][1]; checked += 1
    if abs(y - 5.0) > 0.02:
        bad.append((str(p), y, d['bbox_all']))
print('checked', checked)
print('bad', len(bad))
for item in bad:
    print(item)"
```

For the standard two-printer batch, this should report:

```text
checked 90
bad 0
```

Spot-check a generated Printago package:

```bash
python3 -c "import json, zipfile; p='exports/gcode3mf/p2s/p2s_bin_5x5x18.gcode.3mf'; z=zipfile.ZipFile(p); d=json.loads(z.read('Metadata/project_settings.config')); print(z.testzip()); print(d.get('printer_settings_id')); print(d.get('printer_model')); print('Metadata/plate_1.gcode' in z.namelist())"
```

Expected P2S values:

- `printer_settings_id`: `Bambu Lab P2S 30º Large Part Eject Code`
- `printer_model`: `Bambu Lab P2S`
- embedded G-code: `True`

The P2S preset currently causes Bambu Studio CLI to print
`Invalid T command (T65535)` while slicing. In the tested batch, Bambu Studio
still exited successfully and wrote valid `.gcode.3mf` packages.

If a slice times out, rerun that file or rerun the batch with a larger
`--slice-timeout`, then repeat the 5 mm front-placement audit above. A stale
package can otherwise preserve an older centered plate placement.

## CAD Viewer Thumbnails

The current package step generates simple but size-correct thumbnails with
Pillow. A future polish pass can use CAD Viewer instead:

1. Render each `exports/3mf/bambu/bin_*.3mf` file in CAD Viewer.
2. Capture a consistent square PNG.
3. Inject resized versions into the `.gcode.3mf` package as:
   - `Metadata/plate_1.png`
   - `Metadata/plate_1_small.png`
   - `Metadata/plate_no_light_1.png`
   - `Metadata/top_1.png`
   - `Metadata/pick_1.png`

This would improve visual quality without changing slicing, printer metadata,
or embedded G-code.

## Repository Hygiene

Track the reproducible tooling:

- Python scripts
- CSV/default inputs
- README and future-project notes
- `requirements.txt`
- Bambu template in `templates/bambu/`
- empty profile folder placeholders

Do not commit generated outputs or personal printer exports by default:

- `exports/`
- `work/`
- local printer preset JSONs
- upstream `gridfinity_extended_openscad/`
- old scratch folders

For distribution, prefer GitHub Releases or another artifact store for the
generated 90-file library instead of committing generated files directly.
